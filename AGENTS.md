# AGENTS.md — инструкции для ИИ-агентов

Этот файл — единая точка входа для любого ИИ-агента (Claude Code, Cursor, Aider, Codex и т. д.), работающего в репозитории **LeatherCAM**.

> **Важно:** перед началом любой работы прочитайте этот файл целиком, затем [PLAN.md](PLAN.md) и [PROGRESS.md](PROGRESS.md). После завершения задачи — обновите `PROGRESS.md` и (при необходимости) `PLAN.md`, и сделайте коммит.

---

## 1. О проекте

**LeatherCAM** — кроссплатформенное (Linux + Windows) десктоп-приложение для генерации G-code для CNC 3018 (контроллер GRBL 1.1). Целевой кейс — изготовление клише для тиснения кожи из растровых и векторных изображений.

Полная спецификация и этапы — в [PLAN.md](PLAN.md).

## 5. Принципы кодирования

1. **Прагматизм:** работаем по этапам из `PLAN.md`. Не делаем больше, чем требует текущая задача.
2. **Без преждевременных абстракций.** Три похожих места — повторение лучше, чем фабрика.
3. **Чистая core-логика отделена от UI.** Всё в `leathercam/cam/`, `leathercam/gcode/`, `leathercam/image/`, `leathercam/vector/` должно работать **без Qt** и быть покрыто тестами.
4. **UI-слой (`leathercam/ui/`)** — только обёртки и сигналы. Никакой бизнес-логики.
5. **Без избыточной валидации** на внутренних границах. Валидация — на входе пользователя (UI) и при чтении файлов.
6. **Минимум комментариев.** Только если объясняют *почему*, а не *что*. Никаких "added for task X".
7. **Без emoji** в коде и документации (если пользователь явно не попросит).
8. **G-code строго GRBL 1.1.** Никаких M6, T<n>, G43 и подобных не поддерживаемых команд.
9. **Единицы — миллиметры** (G21). Все внутренние координаты — float в мм.

## 6. Тестирование

- Каждый модуль в `leathercam/` (кроме `ui/`) — должен иметь соответствующий тест в `tests/`.
- Снапшот-тесты G-code: входные параметры → ожидаемый текст. Допуски — через сравнение чисел с точностью 1e-3.
- Перед коммитом запускайте:
  ```bash
  ruff check .
  ruff format --check .
  pytest -q
  ```
- CI прогоняет то же самое на Ubuntu и Windows.

## 7. Работа с PROGRESS.md

`PROGRESS.md` — журнал выполненных задач. Структура:

```
## Этап N. Название

- [x] 2026-05-28 — Задача 1 (commit: abc1234)
- [ ] Задача 2
- [ ] Задача 3
```

**Правила:**
- При **начале** задачи — пометьте её как in-progress (`[~]`) если работаете над ней не в один присест.
- При **завершении** — поставьте `[x]`, дату (формат ISO `YYYY-MM-DD`) и хеш коммита.
- Если задача оказалась неактуальной — `[-]` и пометка "skipped: причина".
- Не удаляйте старые записи — журнал должен расти.

## 9. Git workflow

- **Ветка по умолчанию:** `master`.
- **Feature-ветки** для крупных задач: `feat/stage1-mvp`, `feat/vcarve`, и т. п.
- **Маленькие коммиты** с понятными сообщениями. Лучше 5 коммитов, чем один большой.
- **Conventional Commits**, примеры:
  - `feat(gcode): add GRBL postprocessor with M3/M5 support`
  - `fix(image): correct DPI to mm conversion`
  - `refactor(cam): extract zigzag strategy into separate module`
  - `docs(progress): mark stage 1.3 as done`
  - `test(gcode): add snapshot tests for raster strategy`
  - `chore: add ruff config`
- **Не делайте `--amend`** опубликованных коммитов. Лучше новый коммит.
- **Не делайте `git push --force`** без явной просьбы пользователя.
- **Не пропускайте хуки** (`--no-verify`) без явной просьбы пользователя.

## 11. Чего НЕ делать

- Не создавать markdown-файлы без явной необходимости.
- Не добавлять emoji в код или документацию.
- Не использовать `print` для логирования (только `logging`).
- Не хардкодить пути — использовать `pathlib` и `platformdirs`.
- Не блокировать UI-поток тяжёлыми вычислениями — выносить в `QThread` / `QtConcurrent`.
- Не пушить в удалённый репозиторий без просьбы пользователя.
- Не запускать `rm -rf`, `git reset --hard` и подобные деструктивные команды без явного подтверждения.
